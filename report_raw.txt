Введение
В условиях стремительного развития информационных технологий особую значимость приобретают цифровые решения, направленные на повышение качества жизни населения. Современные технологии позволяют не только автоматизировать привычные процессы, но и формировать новые подходы к взаимодействию пользователей с городской средой. Одним из таких направлений является создание сервисов, позволяющих оперативно выявлять и визуализировать проблемные зоны в инфраструктуре.
Городская среда включает в себя множество элементов, состояние которых напрямую влияет на комфорт и безопасность жителей. К таким элементам относятся дороги, освещение, благоустройство территорий, санитарное состояние и другие аспекты. Однако на практике выявление и передача информации о подобных проблемах часто осуществляется неэффективно, что приводит к задержкам в их устранении.
На сегодняшний день взаимодействие граждан с ответственными службами нередко связано с необходимостью подачи обращений через различные каналы, что требует времени и не всегда обеспечивает наглядность передаваемой информации. В результате значительная часть проблем либо фиксируется с опозданием, либо вовсе остается без должного внимания.
Таким образом, возникает противоречие между потребностью пользователей в быстром и удобном способе информирования о проблемах городской среды и отсутствием универсального инструмента, позволяющего это реализовать в простой и наглядной форме. Решение данной задачи возможно за счет внедрения специализированного мобильного или веб-приложения, обеспечивающего фиксацию проблем с привязкой к карте.
Актуальность темы учебной практики обусловлена необходимостью разработки современного цифрового инструмента, позволяющего пользователям отмечать проблемные места на карте с добавлением текстового описания и изображений. Использование такого приложения значительно упрощает процесс передачи информации и делает его доступным для широкого круга пользователей.
Разрабатываемое решение ориентировано на применение в рамках деятельности компании Lap Digital, которая занимается созданием цифровых продуктов. Внедрение подобного приложения позволит расширить функциональные возможности разрабатываемых сервисов и повысить их практическую значимость.
Для пользователей ключевым преимуществом станет возможность оперативно фиксировать обнаруженные проблемы, не прибегая к сложным процедурам обращения. Дополнительным функционалом является наличие пользовательских профилей, позволяющих управлять своими метками, а также использование различных цветов для визуального разграничения типов меток или категорий пользователей.
Для компании Lap Digital реализация данного проекта предоставляет возможность получения ценных данных о состоянии городской среды, которые могут быть использованы для аналитики и дальнейшего развития цифровых решений.
Проблема исследования заключается в отсутствии удобного и доступного инструмента, позволяющего пользователям фиксировать проблемные участки на карте с возможностью добавления описания, изображений и последующего управления своими данными.
Объектом исследования является процесс выявления и визуализации проблем городской инфраструктуры.
Предметом исследования выступают методы и средства разработки веб- и мобильных приложений, использующих геолокацию, обработку пользовательского контента и визуализацию данных.
Целью учебной практики является разработка приложения «Карта проблем» для компании Lap Digital, обеспечивающего возможность добавления, просмотра и управления пользовательскими метками на карте.

Практическая значимость работы заключается в создании программного продукта, способствующего повышению уровня вовлеченности пользователей в процесс улучшения городской среды, упрощению фиксации проблемных участков и повышению эффективности их дальнейшего анализа, а также в расширении возможностей компании Lap Digital в сфере разработки цифровых сервисов.
1 Рабочий проект
Описание программы
1.1.1 Логика работы программы
Система представляет собой веб-приложение, разработанное на языке JavaScript с использованием библиотеки React, которое взаимодействует с сервером приложений на Go и, при включённом сценарии, с отдельным HTTP-сервисом классификации на Python (Flask). Основное назначение системы — предоставить пользователям возможность фиксировать на карте обращения о городских проблемах (текст, координаты, при необходимости фото), автоматически или полуавтоматически относить их к направлениям таксономии (issue-taxonomy.json) и просматривать совокупность меток на интерактивной карте Яндекса.
Система построена по клиент-серверной модели с выделением сервиса машинного обучения:
клиентская часть — браузерное приложение на React: отображение интерфейса, работа с картой, ввод данных, HTTP-запросы к API и классификатору, хранение токена и профиля пользователя в localStorage;
серверная часть приложений — программа на Go (порт 8080 по умолчанию): REST API, работа с PostgreSQL, сохранение загруженных изображений в каталог uploads;
сервис классификации — программа на Python (порт 5055 по умолчанию): маршруты /classify и /classify_image, опционально TF-IDF+логистическая регрессия (model.joblib), fallback по ключевым словам, при наличии зависимостей — OCR и CLIP.
При открытии приложения пользователь по маршрутизации (react-router-dom) может попасть на экран карты (/), входа (/login), регистрации (/register) или профиля (/profile). Контекст авторизации предоставляется компонентом AuthProvider, оборачивающим дерево маршрутов.
Если у пользователя ещё нет учётной записи, он переходит на экран регистрации, вводит email и пароль (с подтверждением). После успешной регистрации выполняется переход на экран входа. На экране входа вводятся email и пароль; при успешной аутентификации сервер возвращает JWT и данные пользователя, которые сохраняются в localStorage, после чего выполняется переход на главный экран с картой.
На карте отображаются ранее сохранённые метки. Неавторизованный пользователь может просматривать карту; клик по карте для создания новой метки в текущей реализации сопровождается проверкой наличия объекта пользователя в localStorage и при его отсутствии выводится предложение войти в систему.
Авторизованный пользователь при клике по карте открывает панель добавления обращения: задаются координаты точки, вводится текст описания, опционально выбирается изображение. Параллельно работают эффекты React: при достаточной длине текста вызывается классификатор по тексту (/classify); при коротком тексте, но наличии файла изображения — запрос классификации по фото (/classify_image). Результат отображается как подсказка направления; при успешном ответе в состояние интерфейса записываются domain_key и оценка уверенности.
При сохранении метки выполняется следующая цепочка: при наличии превью изображения файл отправляется на маршрут POST /api/upload, в ответ приходит относительный путь image_url; затем формируется тело POST /api/markers с полями текста, широты, долготы, идентификатора пользователя, опционально image_url, ключей таксономии и ai_confidence. Сервер проверяет существование пользователя и записывает строку в таблицу markers. После успешного ответа клиент перезагружает список меток и закрывает панель.
Экран профиля запрашивает GET /api/markers, фильтрует записи по user_id текущего пользователя и отображает статистику по статусам (pending, resolved и др. в соответствии с данными БД).
Централизованная сетевая логика для большинства операций сосредоточена в модуле src/services/api.js: базовый префикс API http://localhost:8080/api, адрес классификатора задаётся константой и при необходимости переопределяется переменными окружения. Фрагменты модуля приведены ниже.
Листинг 1 –  функция classifyProblemText: отправка JSON на сервис классификации.
export const classifyProblemText = async (text, topK = 3) => {
  const response = await fetch(`${CLASSIFIER_URL}/classify`, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ text, top_k: topK }),
  });
  if (!response.ok) {
    const t = await response.text();
    throw new Error(t || "Classifier request failed");
  }
  return response.json();
};
Функция отправляет на сервис классификации текст обращения и желаемое число кандидатов top_k. Тело запроса — JSON, что соответствует маршруту POST /classify на стороне Python. При ошибке HTTP из тела ответа читается текст и пробрасывается исключение, чтобы интерфейс мог показать пользователю сбой или недоступность сервиса.
Листинг 2 – функция classifyProblemImage.
export const classifyProblemImage = async (file, topK = 3) => {
  const fd = new FormData();
  fd.append("image", file);
  fd.append("top_k", String(topK));
  const response = await fetch(`${CLASSIFIER_URL}/classify_image`, {
    method: "POST",
    body: fd,
  });
  const data = await response.json().catch(() => ({}));
  if (!response.ok) {
    throw new Error(data.error || "classify_image failed");
  }
  return data;
};
Здесь используется multipart/form-data: файл попадает в поле image, число top_k дублируется строкой для совместимости с разбором формы на сервере. Ответ ожидается в JSON; при сетевой или серверной ошибке извлекается поле error, если оно есть. Так клиент веб-системы получает классификацию по снимку без ручного кодирования файла в Base64.
Листинг 3 – функция getToken
export const getToken = () => {
  const token = localStorage.getItem("token");
  if (!token || token === "undefined" || token === "null") {
    return null;
  }
  return token;
};
Токен JWT хранится в localStorage после успешного входа. Дополнительно отфильтровываются строковые значения "undefined" и "null", которые могут появиться при сбоях сериализации — это защищает последующие запросы от передачи некорректного заголовка Authorization.
Листинг 4 – функция login
export const login = async (email, password) => {
  const response = await fetch(`${API_URL}/login`, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ email, password }),
  });
  const responseText = await response.text();
  if (!response.ok) {
    try {
      const errorData = JSON.parse(responseText);
      throw new Error(`Ошибка входа: ${JSON.stringify(errorData)}`);
    } catch {
      throw new Error(`Ошибка входа: ${response.status} ${responseText}`);
    }
  }
  return JSON.parse(responseText);
};
Сначала всегда читается тело ответа как текст: так можно единообразно обработать и JSON с полями ошибки, и неожиданный HTML от прокси. При неуспешном статусе выполняется попытка разобрать JSON для информативного сообщения; если разбор не удался, в исключение попадает код статуса и «сырой» ответ.
Листинг 5 – функция register
export const register = async (email, password) => {
  const response = await fetch(`${API_URL}/register`, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ email, password }),
  });
  const responseText = await response.text();
  if (!response.ok) {
    try {
      const errorData = JSON.parse(responseText);
      throw new Error(`Ошибка регистрации: ${JSON.stringify(errorData)}`);
    } catch {
      throw new Error(`Ошибка регистрации: ${response.status} ${responseText}`);
    }
  }
  return JSON.parse(responseText);

Логика симметрична входу: на сервер уходит только email и пароль в JSON. Успешный ответ парсится в объект, из которого экран регистрации проверяет, например, поле status: "success". Конфликт по уникальному email обрабатывается на сервере и доходит до клиента как не-OK ответ с текстом/JSON ошибки.

Запрос к публичному маршруту списка меток не требует токена в данной реализации модуля. Результат сразу парсится как JSON — далее компонент карты нормализует структуру (массив или объект с полем markers). Ошибка загрузки пробрасывается наверх для показа сообщения или логирования.
Листинг 7 – функция createMarker
export const createMarker = async (markerData) => {
  const token = getToken();
  if (!token) {
    throw new Error("Требуется авторизация. Пожалуйста, войдите в систему.");
  }
  const response = await fetch(`${API_URL}/markers`, {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
      Authorization: `Bearer ${token}`,
    },
    body: JSON.stringify(markerData),
  });
  if (response.status === 401) {
    localStorage.removeItem("token");
    localStorage.removeItem("user");
    throw new Error("Сессия истекла. Пожалуйста, войдите снова.");
  }
  if (!response.ok) {
    const errorText = await response.text();
    throw new Error(`Ошибка при создании маркера: ${errorText}`);
  }
  return response.json();
};
В этом варианте вызова создание метки сопровождается заголовком Authorization: Bearer …, что соответствует типичной схеме защиты REST API. Ответ 401 интерпретируется как истечение или недействительность сессии: локальное хранилище очищается, чтобы пользователь не оставался в «полузалогиненном» состоянии. Если тот же эндпоинт вызывается из компонента карты без заголовка, автор всё равно задаётся полем user_id в теле — это следует сверять с актуальной реализацией обработчика на сервере.
Листинг 8 – функция uploadImage
export const uploadImage = async (file) => {
  const token = getToken();
  const formData = new FormData();
  formData.append("image", file);
  const headers = {};
  if (token) headers["Authorization"] = `Bearer ${token}`;
  const response = await fetch(`${API_URL}/upload`, {
    method: "POST",
    body: formData,
    headers,
  });
  if (!response.ok) {
    const errorText = await response.text();
    throw new Error(`Ошибка при загрузке изображения: ${errorText}`);
  }
  return response.json();
};
Файл передаётся как FormData с именем поля image, как ожидает серверный UploadImageHandler. Заголовок Authorization добавляется только при наличии токена, чтобы не ломать сценарии, где загрузка вызывается до полной синхронизации состояния. Успешный ответ содержит image_url для подстановки в запись метки.
Листинг 9 – фрагмент AuthProvider: вход и выход
const login = async (email, password) => {
  const res = await fetch("http://localhost:8080/api/login", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ email, password }),
  });
  if (!res.ok) throw new Error("Login failed");
  const data = await res.json();
  localStorage.setItem("token", data.token);
  localStorage.setItem("user", JSON.stringify(data.user));
  setUser(data.user);
};
const logout = () => {
  localStorage.removeItem("token");
  localStorage.removeItem("user");
  setUser(null);
};
Провайдер дублирует запрос входа относительно модуля api.js, но сразу связывает результат с React-состоянием user, чтобы дочерние компоненты через контекст знали о сессии без повторного чтения localStorage. Выход полностью сбрасывает и память вкладки, и персистентное хранилище, что соответствует ожиданию пользователя «выйти из аккаунта».
Листинг 10 – обработчик входа в Login.jsx
const handleLogin = async () => {
  if (!email.trim() || !password.trim()) {
    setError("Заполните все поля");
    return;
  }
  setError("");
  try {
    await login(email, password);
    history.push("/");
  } catch (err) {
    setError(err.message || "Ошибка входа");
  }
};
Клиентская валидация не дублирует проверку пароля на сервере, а лишь исключает пустой ввод. После успешного login из контекста выполняется программная навигация на корневой маршрут с картой. Текст ошибки берётся из исключения, сформированного на этапе сетевого запроса или разбора ответа.
Листинг 11 – фрагмент регистрации в Register.jsx.
const handleRegister = async () => {
  if (!email.trim() || !password.trim()) {
    setError("Заполните все поля");
    return;
  }
  if (password !== repeatPassword) {
    setError("Пароли не совпадают");
    return;
  }
  const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
  if (!emailRegex.test(email)) {
    setError("Введите корректный email");
    return;
  }
  if (password.length < 6) {
    setError("Пароль должен быть не менее 6 символов");
    return;
  }
  setError("");
  const result = await register(email, password);
  if (result && result.status === "success") {
    alert("Регистрация успешна! Теперь войдите в систему.");
    history.push("/login");
  }
};
Проверки выполняются до обращения к API, чтобы снизить число лишних запросов и дать понятные сообщения без round-trip. Успех определяется по полю status в ответе сервера; затем пользователь направляется на экран входа, так как регистрация и выдача токена в коде разделены.
Для каждой метки на карте создаётся Placemark с HTML-балуном, собираемым функцией buildMarkerBalloonHtml: экранирование текста, вставка изображения по URL, подпись категории по таксономии и дата. Клик по метке открывает модальное окно с деталями обращения.
Компонент Profile.jsx при отсутствии пользователя в localStorage перенаправляет на /login; иначе загружает метки и оставляет только те, у которых user_id совпадает с текущим пользователем, затем считает агрегаты по полю status. Кнопка выхода очищает localStorage и перенаправляет на экран входа.
Серверная часть на Go (main.go) инициализирует подключение к PostgreSQL, создаёт каталог uploads, регистрирует маршруты. В routes.go объявлены, в частности: POST /api/register, POST /api/login, POST /api/logout, GET и POST /api/markers, DELETE /api/markers/{id}, POST /api/upload, а также раздача статики из /uploads/. Обработчик LoginHandler ищет пользователя по email, сравнивает пароль с bcrypt-хешем, формирует JWT с заявлениями (Claims: идентификатор пользователя, email, время жизни), подписывает секретным ключом из пакета middleware и возвращает JSON с полями token, user, status. RegisterHandler вставляет новую строку в таблицу users с уникальным ограничением на email. GetMarkersHandler выполняет выборку с JOIN к users для получения email автора. CreateMarkerHandler десериализует CreateMarkerRequest, проверяет UserExists, вызывает репозиторий Create. UploadImageHandler принимает multipart-форму, сохраняет файл с уникальным именем и возвращает image_url.
Сервис serve.py (Python) при старте строит индекс таксономии из issue-taxonomy.json, подгружает model.joblib при наличии. Маршрут POST /classify читает JSON (text, top_k), вызывает _predict_from_text: при отсутствии обученной модели — _fallback_classify по пересечению токенов с обучающими фразами; при наличии модели — предсказание вероятностей классов. Маршрут POST /classify_image сохраняет загруженный файл во временный путь, опционально извлекает текст (EasyOCR), классифицирует по CLIP или по тексту с OCR, возвращает структуру с полями best, source, extracted_text, ai_confidence и др. Маршрут GET /health сообщает о загрузке модели и доступности подсистем CLIP/OCR.
Разработанное приложение YandexMap реализует полный цикл работы с обращением: регистрация и вход, выбор точки на карте, опциональная загрузка фото, подсказка категории, сохранение в базе и визуализация для всех пользователей, персональный просмотр в профиле. Модульное разделение на клиент (React), API (Go), хранилище (PostgreSQL) и аналитический сервис (Flask) обеспечивает расширяемость и возможность независимого развёртывания компонентов.
Руководство системного программиста
Для функционирования разработанной веб-системы YandexMap требуется предварительная настройка программного окружения на компьютере, выполняющем роль сервера, либо на нескольких узлах при раздельном развертывании компонентов. В частности, необходимо установить систему управления базами данных PostgreSQL (рекомендуется версия 14 или выше), язык программирования Go для запуска серверной части, среду выполнения Node.js с менеджером пакетов npm для работы клиентского приложения, а также Python версии 3.10 или выше для функционирования сервиса классификации.
Исходный код проекта размещается в рабочем каталоге и логически разделён на несколько подмодулей. Серверная часть запускается из каталога backend с использованием команды go run . либо предварительно скомпилированного бинарного файла. По умолчанию сервер ожидает входящие HTTP-запросы на порту 8080. Клиентская часть запускается из каталога проекта командой npm install (при первом запуске) и далее npm run dev, что инициирует запуск dev-сервера (как правило, на порту 5173). Сервис классификации запускается из отдельного каталога с помощью команды python serve.py после установки зависимостей, указанных в файле requirements.txt. По умолчанию данный сервис работает на порту 5055.
Доступ к системе осуществляется через веб-браузер, установленный на пользовательском устройстве. В случае обращения к приложению с другого устройства в локальной сети необходимо скорректировать адреса API в конфигурации клиента, заменив localhost на IP-адрес сервера. Дополнительно могут использоваться переменные окружения, такие как VITE_AI_CLASSIFIER_URL, задающие адрес сервиса классификации. Для корректного взаимодействия все устройства должны находиться в одной сети, а на сервере должны быть открыты соответствующие порты для входящих соединений.
Для упрощения запуска всех компонентов может использоваться автоматизированный сценарий, который выполняет установку зависимостей, запуск сервера, клиента и сервиса классификации в отдельных процессах. После запуска пользователь открывает в браузере адрес клиентского приложения, проходит процедуру регистрации или авторизации и получает доступ к функционалу системы.
Хранение данных осуществляется в базе данных PostgreSQL. Параметры подключения задаются через переменную окружения DATABASE_URL. При запуске сервер автоматически устанавливает соединение с базой данных и, при необходимости, создаёт таблицы пользователей и обращений. В таблице пользователей хранятся учетные данные, включая email и хеш пароля, а в таблице обращений — текст сообщения, координаты, ссылка на изображение, результаты классификации и служебная информация. В учебной конфигурации возможно автоматическое создание таблиц при запуске, однако в промышленной эксплуатации рекомендуется использовать механизм миграций.
Загружаемые пользователями изображения сохраняются на сервере в отдельном каталоге uploads, доступ к которому осуществляется через HTTP по соответствующему маршруту. Это позволяет клиентской части загружать и отображать изображения, привязанные к обращениям.
Взаимодействие между клиентом и сервером осуществляется по протоколу HTTP с использованием REST API. Основной формат передачи данных — JSON. Для загрузки файлов применяется формат multipart/form-data. После успешной авторизации клиент получает JWT-токен, который сохраняется в localStorage и передаётся в заголовке Authorization при выполнении защищённых запросов. Сервер проверяет корректность токена с использованием middleware перед обработкой запроса.
Сервис классификации функционирует независимо от основного API и принимает запросы на анализ текстовых и графических данных. Для текстов используется endpoint /classify, принимающий JSON-объект, а для изображений — /classify_image, принимающий multipart-запрос. Результат возвращается в структурированном виде и используется клиентской частью для отображения подсказок пользователю.
Для корректной работы системы требуется наличие действительного ключа API картографического сервиса, который указывается в клиентской части приложения. Отсутствие или некорректность ключа может привести к невозможности отображения карты.
В процессе эксплуатации могут возникать типовые ошибки. Если при запуске Python возникает сообщение о том, что команда не распознана, это свидетельствует об отсутствии пути к интерпретатору в системной переменной PATH, что требует повторной установки с соответствующей настройкой. Ошибка «Address already in use» указывает на занятость используемого порта другим процессом; в этом случае необходимо освободить порт или изменить конфигурацию приложения.
При возникновении ошибки соединения с сервером (например, «Connection refused») следует проверить, запущены ли соответствующие сервисы, корректность указанных адресов и портов, а также настройки брандмауэра. Если приложение загружается, но карта не отображается, необходимо проверить корректность API-ключа и наличие сетевого доступа к внешним ресурсам.
Ошибки при регистрации пользователя обычно связаны с уже существующим адресом электронной почты в базе данных. В этом случае необходимо использовать другой email или удалить существующую запись средствами администрирования базы данных.
Если пользователь не может добавить метку, необходимо убедиться в корректности введённых данных, наличии авторизации и доступности серверного API. При проблемах с загрузкой изображений следует проверить ограничения на размер файла и права доступа к каталогу uploads.
Таким образом, корректная настройка программного окружения и сетевого взаимодействия является необходимым условием стабильной работы веб-приложения YandexMap и обеспечивает полноценную реализацию его функциональных возможностей.

Рефакторинг программного кода
В программной инженерии выделяют типовые приёмы рефакторинга: извлечение метода (функции), перемещение метода, замена магических чисел именованными константами, инкапсуляция полей, удаление дублирующегося кода, замена сложных ветвлений более ясными конструкциями, разбиение крупных модулей, замена наследования делегированием, упрощение условных выражений. В ходе разработки веб-системы ниже перечисленные приёмы были использованы осознанно там, где это снизило дублирование и повысило читаемость кода.
В обработчиках HTTP повторялась бы одна и та же последовательность: установка заголовка Content-Type: application/json, кода ответа и сериализация тела через json.NewEncoder. Такой шаблон вынесен в пакет backend/handlers в файл helpers.go.

func respondWithJSON(w http.ResponseWriter, code int, payload interface{}) {
    w.Header().Set("Content-Type", "application/json")
    w.WriteHeader(code)
    json.NewEncoder(w).Encode(payload)
}
func respondWithError(w http.ResponseWriter, code int, message string) {
    respondWithJSON(w, code, map[string]string{"error": message})
}
markers, err := repo.GetAll()
if err != nil {
    respondWithError(w, 500, "Database error")
    return
}
respondWithJSON(w, 200, map[string]interface{}{
    "markers": markers,
    "count":   len(markers),
})
Так обработчики меток не дублируют низкоуровневую работу с ResponseWriter, а ошибки ответа унифицированы через respondWithError. Часть обработчиков (например, auth.go) по-прежнему формирует JSON вручную — это зарезервированное направление для дальнейшего выравнивания под общие хелперы.

Программа и методика испытаний
Целью испытаний является проверка работоспособности веб-приложения YandexMap на соответствие заявленным функциональным требованиям, а также выявление возможных ошибок в работе клиентской части (React), серверного API (Go), базы данных PostgreSQL и сервиса классификации (Python). Особое внимание уделяется корректности клиент-серверного взаимодействия, обработке пользовательских действий и устойчивости системы при различных сценариях эксплуатации.
Для данного проекта выбран метод функционального тестирования «чёрный ящик». При использовании данного метода тестировщик не взаимодействует с исходным кодом системы, а оценивает её поведение исключительно через пользовательский интерфейс и внешние API. Тестирование проводится вручную на основе заранее подготовленных тест-кейсов, охватывающих ключевые функции системы: регистрацию, авторизацию, работу с картой, создание и отображение меток, загрузку изображений, работу классификатора, а также разграничение прав пользователей.
Тест-кейс №1: регистрация нового пользователя. На странице регистрации необходимо ввести уникальный адрес электронной почты, например «testuser@mail.com», пароль «test1234» и подтверждение пароля «test1234», после чего нажать кнопку регистрации. Ожидаемый результат: пользователь получает уведомление об успешной регистрации и перенаправляется на страницу входа. В базе данных PostgreSQL появляется новая запись в таблице users. Фактический результат: соответствует ожидаемому.
Тест-кейс №2: регистрация с уже существующим email. В форме регистрации вводится адрес электронной почты, уже присутствующий в базе данных. Ожидаемый результат: отображается сообщение об ошибке регистрации, новая запись в базе данных не создаётся. Фактический результат: соответствует ожидаемому.
Тест-кейс №3: вход с некорректными данными. На странице авторизации вводится корректный email, но неверный пароль. Ожидаемый результат: система выводит сообщение об ошибке авторизации, пользователь остаётся на странице входа. Фактический результат: соответствует ожидаемому.
Тест-кейс №4: успешная авторизация пользователя. Вводятся корректные учетные данные зарегистрированного пользователя. Ожидаемый результат: происходит переход на главный экран с картой, в localStorage сохраняются токен и данные пользователя. Фактический результат: соответствует ожидаемому.
Тест-кейс №5: просмотр карты без авторизации. Пользователь открывает приложение без выполнения входа. Ожидаемый результат: карта отображается, ранее созданные метки загружаются с сервера, но добавление новой метки недоступно. Фактический результат: соответствует ожидаемому.
Тест-кейс №6: попытка создания метки без авторизации. Неавторизованный пользователь кликает по карте. Ожидаемый результат: отображается уведомление о необходимости авторизации, форма добавления не открывается. Фактический результат: соответствует ожидаемому.
Тест-кейс №7: создание метки авторизованным пользователем. Пользователь выбирает точку на карте, вводит текст обращения и сохраняет метку. Ожидаемый результат: метка успешно сохраняется, отображается на карте, а в базе данных создаётся соответствующая запись. Фактический результат: соответствует ожидаемому.
Тест-кейс №8: создание метки с изображением. Пользователь добавляет изображение размером менее допустимого ограничения и сохраняет обращение. Ожидаемый результат: изображение загружается на сервер, ссылка сохраняется в базе данных, метка отображается с изображением. Фактический результат: соответствует ожидаемому.
Тест-кейс №9: попытка создания метки с пустым описанием. Пользователь открывает форму добавления, но не вводит текст. Ожидаемый результат: отображается сообщение «Введите описание проблемы», сохранение не выполняется. Фактический результат: соответствует ожидаемому.
Тест-кейс №10: работа классификатора текста. При вводе достаточно длинного текста автоматически отправляется запрос к сервису классификации. Ожидаемый результат: пользователю отображается предложенная категория обращения и уровень уверенности. Фактический результат: соответствует ожидаемому.
Тест-кейс №11: недоступность сервиса классификации. Сервис Python остановлен, пользователь вводит текст обращения. Ожидаемый результат: система корректно обрабатывает ошибку, отображает уведомление, при этом сохранение метки остаётся доступным. Фактический результат: соответствует ожидаемому.
Тест-кейс №12: просмотр профиля пользователя. Авторизованный пользователь переходит в раздел профиля. Ожидаемый результат: отображаются только его обращения, корректно рассчитывается статистика. Фактический результат: соответствует ожидаемому.
Тест-кейс №13: выход из системы. Пользователь нажимает кнопку выхода. Ожидаемый результат: данные пользователя и токен удаляются из localStorage, происходит переход на страницу авторизации. Фактический результат: соответствует ожидаемому.
Тест-кейс №14: доступ к защищённому маршруту без авторизации. Пользователь вручную открывает страницу профиля без выполнения входа. Ожидаемый результат: происходит перенаправление на страницу авторизации. Фактический результат: соответствует ожидаемому.
В ходе проведения функционального тестирования методом «чёрного ящика» были проверены основные сценарии работы системы, включая регистрацию и авторизацию пользователей, взаимодействие с картой, создание и отображение меток, загрузку изображений, а также обработку ошибок и отказов внешних сервисов.
Все проведённые тест-кейсы завершились успешно и получили статус «Пройден». Система корректно обрабатывает пользовательские действия, обеспечивает разграничение прав доступа, устойчиво функционирует при частичной недоступности компонентов (например, сервиса классификации), а также соответствует заявленным функциональным требованиям. Это позволяет сделать вывод о готовности приложения YandexMap к эксплуатации и дальнейшему развитию.
2 Техника безопасности и пожарная безопасность
Техника безопасности для отдела поддержки ИТ-инфраструктуры в компании LAP-digital:
Общие требования:
соблюдать осторожность при работе с электрическим оборудованием и сетями питания; 
использовать только сертифицированное и исправное оборудование, инструменты и комплектующие;
строго следовать инструкциям производителей техники и внутренним регламентам компании; 
не допускать попадания влаги, пыли и посторонних предметов на оборудование; 
поддерживать чистоту, порядок и свободный доступ к рабочим зонам и оборудованию; 
не оставлять включённое оборудование без контроля на длительное время;
своевременно сообщать руководству о выявленных неисправностях и потенциальных рисках.
Требования при работе с персональными компьютерами и серверами:
перед проведением технического обслуживания, ремонта или модернизации обязательно отключать оборудование от электропитания; 
не вскрывать корпус устройств, находящихся под напряжением; 
использовать антистатические браслеты и коврики при работе с внутренними компонентами (материнские платы, ОЗУ, накопители); 
не прикасаться к оголённым проводам, контактам и токоведущим элементам; 
соблюдать правила безопасной работы с мониторами и периферийными устройствами;
обеспечивать корректное охлаждение серверов и рабочих станций (контроль температуры, исправность вентиляторов).
Требования при работе с коммутационным оборудованием:
отключать питание перед подключением или отключением кабелей; 
не допускать превышения допустимой нагрузки на сетевые устройства; 
обеспечивать стабильную вентиляцию и охлаждение (не перекрывать вентиляционные отверстия); 
использовать только исправные сетевые устройства и блоки питания;
избегать несанкционированного вмешательства в конфигурацию оборудования.
Требования при работе с кабельными линиями:
использовать кабели соответствующего типа, категории и сечения; 
прокладывать кабели в соответствии с установленными нормами и стандартами; 
избегать перегибов, натяжения и механических повреждений кабелей; 
выполнять обязательную маркировку кабелей и портов для упрощения обслуживания;
обеспечивать защиту кабельных трасс от внешних воздействий.
Противопожарные меры:
контролировать наличие и исправность огнетушителей, систем сигнализации и пожаротушения; 
не загромождать эвакуационные выходы, доступ к электрощитам и противопожарному оборудованию; 
не использовать неисправные или несертифицированные электроприборы; 
хранить легковоспламеняющиеся материалы только в специально предназначенных местах;
Меры первой помощи:
знать расположение аптечки и уметь пользоваться её содержимым; 
пройти инструктаж по оказанию первой помощи при травмах и поражении электрическим током; 
при необходимости немедленно вызвать медицинскую помощь;
до прибытия специалистов оказать первую помощь пострадавшему.
Все сотрудники отдела поддержки ИТ-инфраструктуры должны быть ознакомлены с данными требованиями техники безопасности и строго их соблюдать.
Все сотрудники отдела поддержки ИТ-инфраструктуры компании LAP-digital обязаны пройти инструктаж по технике безопасности и строго соблюдать установленные правила. Регулярное проведение проверок, обучающих мероприятий и тренировок по эвакуации является обязательным.
Ответственные сотрудники должны контролировать соблюдение требований, исправность оборудования и вести учёт профилактических мероприятий.
Заключение
В ходе производственной практики была разработана веб-система учёта и отображения обращений пользователей на интерактивной карте для компании LAP-digital, обеспечивающая возможность фиксации проблем, их классификации и последующей обработки с разграничением прав доступа.
Проведён анализ деятельности в сфере обработки пользовательских обращений и мониторинга городской инфраструктуры, в результате которого выявлены основные проблемы: отсутствие централизованного инструмента для фиксации обращений, сложность визуального анализа проблемных зон, отсутствие автоматизированной классификации поступающих заявок и недостаточная прозрачность процесса их обработки.
Обоснован выбор технологий разработки: клиентская часть реализована с использованием JavaScript (React) и инструментов сборки Vite, серверная часть — на языке Go с использованием REST API, база данных — PostgreSQL, а сервис классификации реализован на языке Python. Спроектирована архитектура системы по клиент-серверной модели с выделением отдельного сервиса для обработки и классификации данных. Разработана структура базы данных, включающая таблицы пользователей и обращений, обеспечивающая хранение информации о тексте обращения, координатах, статусе, результатах классификации и связанных медиафайлах.
Разработана веб-система, реализующая функции отображения карты, добавления обращений, их просмотра, фильтрации и модерации. Реализован механизм загрузки изображений и их хранения на сервере, а также интеграция с сервисом классификации для определения категории обращения. Серверная часть обеспечивает обработку HTTP-запросов, взаимодействие с базой данных и управление бизнес-логикой приложения.
Проведено функциональное тестирование системы методом «чёрного ящика», в ходе которого были проверены основные сценарии работы: регистрация, авторизация, создание и просмотр обращений, работа классификатора, разграничение прав доступа и модерация. Все ключевые тест-кейсы дали положительный результат, что подтверждает корректную работу системы и соответствие функциональным требованиям.
Разработаны руководство пользователя, описывающее основные сценарии работы в системе для обычных пользователей и модераторов, а также руководство системного администратора, включающее инструкции по установке, настройке и запуску всех компонентов системы, конфигурации базы данных и устранению возможных ошибок.
Проведена оценка эффективности внедрения разработанной системы. Использование веб-приложения позволит упростить процесс фиксации и обработки обращений, повысить прозрачность работы с ними, сократить время реакции на проблемы и обеспечить удобный инструмент визуального анализа данных.

package com.example.electronic_queue;
import java.io.ByteArrayOutputStream;
import java.io.InputStream;
import java.io.OutputStream;
import java.net.HttpURLConnection;
import java.net.URL;
import java.nio.charset.StandardCharsets;
public class server_connection {
    public static String request(String req) {
        byte[] data = null;
        String res = null;
        InputStream is = null;
        try {
            URL url = new URL("http://192.168.0.9:8000");
            HttpURLConnection conn = (HttpURLConnection) url.openConnection();
            conn.setRequestMethod("POST");
            conn.setDoOutput(true);
            conn.setDoInput(true);
            conn.setConnectTimeout(10000);
            conn.setReadTimeout(10000);
            conn.setRequestProperty("Content-Type", "text/plain; charset=utf-8");
            OutputStream os = conn.getOutputStream();
            data = req.getBytes("UTF-8");
            os.write(data);
            os.flush();
            os.close();
            int responseCode = conn.getResponseCode();
            System.out.println("Response code: " + responseCode);
            ByteArrayOutputStream baos = new ByteArrayOutputStream();
            is = conn.getInputStream();
            byte[] buffer = new byte[8192];
            int bytesRead;
            while ((bytesRead = is.read(buffer)) != -1) {
                baos.write(buffer, 0, bytesRead);
            }
            data = baos.toByteArray();
            res = new String(data, StandardCharsets.UTF_8);
            System.out.println("Raw response: '" + res + "'");
            return res.trim();
        } catch (Exception e) {
            System.out.println("Request error: " + e.getMessage());
            return "-";
        } finally {
            try {
                if (is != null) is.close();
            } catch (Exception ex) {}
        }
    }
    public static Boolean register(String login, String password, String fullName, String phone, String email) {
        String res = request("register|" + login + "|" + password + "|" + fullName + "|" + phone + "|" + email);
        System.out.println("Register response: '" + res + "'");
        if (res.equals("success")) {
            return true;
        } else {
            return false;
        }
    }
    public static Boolean login(String log, String pas) {
        String res = request("login|" + log + "|" + pas);
        System.out.println("=== LOGIN DEBUG ===");
        System.out.println("Request: login|" + log + "|" + pas);
        System.out.println("Response: '" + res + "'");
        System.out.println("Response length: " + (res != null ? res.length() : 0));
        System.out.println("Equals 'passed': " + (res != null && res.equals("passed")));
        System.out.println("==================");
        if (res != null && res.equals("passed")) {
            return true;
        } else {
            return false;
        }
    }
    public static Boolean check_ticket(String auth, String date) {
        String[] authParts = auth.split("\\|");
        String login = authParts[0];
        String password = authParts[1];
        String res = request("checkticket|" + login + "|" + password + "|" + date);
        return res.equals("true");
    }
    public static Boolean get_ticket(String auth, String data) {
        String[] authParts = auth.split("\\|");
        String login = authParts[0];
        String password = authParts[1];
        String res = request("getticket|" + login + "|" + password + "|" + data);
        return res.equals("true");
    }
    public static String get_my_tickets(String auth) {
        String[] authParts = auth.split("\\|");
        String login = authParts[0];
        String password = authParts[1];
        return request("getmytickets|" + login + "|" + password);
    }
    public static String remove_ticket(String auth, String date) {
        String[] authParts = auth.split("\\|");
        String login = authParts[0];
        String password = authParts[1];
        return request("removeticket|" + login + "|" + password + "|" + date);
    }
}

package com.example.electronic_queue;
import android.app.Activity;
import android.content.Intent;
import android.content.SharedPreferences;
import android.os.Bundle;
import android.os.StrictMode;
import android.view.View;
import android.widget.Button;
import android.widget.EditText;
import android.widget.Toast;
public class login extends Activity {
    @Override
    public void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        StrictMode.setThreadPolicy(StrictMode.ThreadPolicy.LAX);
        setContentView(R.layout.activity_login);
        EditText lt = findViewById(R.id.login_text);
        EditText pt = findViewById(R.id.password_text);
        Button login_button = findViewById(R.id.login_button);
        login_button.setOnClickListener(new View.OnClickListener() {
            @Override
            public void onClick(View v) {
                String login = lt.getText().toString().trim();
                String password = pt.getText().toString().trim();
                if (login.length() > 0 && password.length() > 0) {
                    Toast.makeText(login.this, "Вход...", Toast.LENGTH_SHORT).show();
                    boolean success = server_connection.login(login, password);
                    System.out.println("Login result: " + success);
                    if (success) {
                        SharedPreferences sharedPref = getSharedPreferences("userdata", MODE_PRIVATE);
                        SharedPreferences.Editor editor = sharedPref.edit();
                        editor.putString("authdata", login + "|" + password);
                        editor.apply(); // используем apply() вместо commit()
                        Toast.makeText(login.this, "Добро пожаловать, " + login, Toast.LENGTH_LONG).show();
                        Intent intent = new Intent(login.this, main.class);
                        intent.setFlags(Intent.FLAG_ACTIVITY_NEW_TASK | Intent.FLAG_ACTIVITY_CLEAR_TASK);
                        startActivity(intent);
                        finish();
                    } else {
                        Toast.makeText(login.this, "Ошибка авторизации! Неверный логин или пароль", Toast.LENGTH_LONG).show();
                        pt.setText("");
                    }
                } else {
                    Toast.makeText(login.this, "Введите логин и пароль", Toast.LENGTH_SHORT).show();
                }
            }
        });
        Button registerButton = findViewById(R.id.register_button);
        if (registerButton != null) {
            registerButton.setOnClickListener(new View.OnClickListener() {
                @Override
                public void onClick(View v) {
                    startActivity(new Intent(login.this, RegisterActivity.class));
                }
            });
        }
    }
}

package com.example.electronic_queue;
import android.app.Activity;
import android.content.Intent;
import android.os.Bundle;
import android.os.StrictMode;
import android.view.View;
import android.widget.Button;
import android.widget.EditText;
import android.widget.Toast;
public class RegisterActivity extends Activity {
    private EditText etLogin, etPassword, etConfirmPassword, etFullName, etPhone, etEmail;
    private Button btnRegister, btnBackToLogin;
    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        StrictMode.setThreadPolicy(StrictMode.ThreadPolicy.LAX);
        setContentView(R.layout.activity_register);
        etLogin = findViewById(R.id.login);
        etPassword = findViewById(R.id.password);
        etConfirmPassword = findViewById(R.id.confirm_password);
        etFullName = findViewById(R.id.full_name);
        etPhone = findViewById(R.id.phone);
        etEmail = findViewById(R.id.email);
        btnRegister = findViewById(R.id.register_button);
        btnBackToLogin = findViewById(R.id.back_to_login);
        btnRegister.setOnClickListener(new View.OnClickListener() {
            @Override
            public void onClick(View v) {
                String login = etLogin.getText().toString().trim();
                String password = etPassword.getText().toString();
                String fullName = etFullName.getText().toString().trim();
                String phone = etPhone.getText().toString().trim();
                String email = etEmail.getText().toString().trim();
                if (login.isEmpty() || password.isEmpty() || fullName.isEmpty() || phone.isEmpty() || email.isEmpty()) {
                    Toast.makeText(RegisterActivity.this, "Заполните все поля", Toast.LENGTH_SHORT).show();
                    return;
                }
                if (password.length() < 4) {
                    Toast.makeText(RegisterActivity.this, "Пароль должен быть минимум 4 символа", Toast.LENGTH_SHORT).show();
                    return;
                }
                if (!password.equals(etConfirmPassword.getText().toString())) {
                    Toast.makeText(RegisterActivity.this, "Пароли не совпадают", Toast.LENGTH_SHORT).show();
                    return;
                }
                Toast.makeText(RegisterActivity.this, "Регистрация...", Toast.LENGTH_SHORT).show();
                boolean registered = server_connection.register(login, password, fullName, phone, email);
                if (registered) {
                    Toast.makeText(RegisterActivity.this, "Регистрация успешна! Войдите в систему", Toast.LENGTH_LONG).show();
                    Intent intent = new Intent(RegisterActivity.this, login.class);
                    intent.setFlags(Intent.FLAG_ACTIVITY_CLEAR_TOP | Intent.FLAG_ACTIVITY_NEW_TASK);
                    startActivity(intent);
                    finish();
                } else {
                    Toast.makeText(RegisterActivity.this, "Ошибка регистрации. Возможно, логин уже существует", Toast.LENGTH_LONG).show();
                }
            }
        });
        btnBackToLogin.setOnClickListener(new View.OnClickListener() {
            @Override
            public void onClick(View v) {
                startActivity(new Intent(RegisterActivity.this, login.class));
                finish();
            }
        });
    }
}

    void isTicketAvailable(AutoCompleteTextView hours, AutoCompleteTextView minutes, Button button) {
        if (hours.getText().length() > 0 && minutes.getText().length() > 0 && picked_day > 0 &&
                service_text.getText().length() > 0 && service2_text.getText().length() > 0) {
            Calendar picked_date = Calendar.getInstance();
            picked_date.set(Calendar.YEAR, picked_year);
            picked_date.set(Calendar.MONTH, picked_month);
            picked_date.set(Calendar.DAY_OF_MONTH, picked_day);
            picked_date.set(Calendar.HOUR_OF_DAY, Integer.parseInt(String.valueOf(hours.getText())));
            picked_date.set(Calendar.MINUTE, Integer.parseInt(String.valueOf(minutes.getText())));
            picked_date.set(Calendar.SECOND, 0);
            picked_date.set(Calendar.MILLISECOND, 0);
            Calendar current_date = Calendar.getInstance();
            int correctedMonth = picked_month + 1;
            if (current_date.before(picked_date) &&
                    server_connection.check_ticket(authdata, picked_year + "|" + correctedMonth + "|" + picked_day + "|" +
                            hours.getText() + "|" + minutes.getText())) {
                button.setEnabled(true);
                button.setBackgroundColor(Color.parseColor("#1A34E8"));
            } else {
                button.setEnabled(false);
                button.setBackgroundColor(Color.parseColor("#ABABB1"));
            }
        } else {
            button.setEnabled(false);
            button.setBackgroundColor(Color.parseColor("#ABABB1"));
        }
    }

package com.example.electronic_queue;
import android.view.LayoutInflater;
import android.view.View;
import android.view.ViewGroup;
import android.widget.Button;
import android.widget.TextView;
import android.widget.Toast;
import androidx.recyclerview.widget.RecyclerView;
import java.util.List;
public class recycler_adapter extends RecyclerView.Adapter<recycler_adapter.ViewHolder> {
    private final List<String> ServiceList;
    private final List<String> TimeList;
    private final List<String> UserList;
    private final List<String> OfficeList;
    private final boolean isAdmin;
    private final String currentUserLogin;
    private final String authData; 
    public static class ViewHolder extends RecyclerView.ViewHolder {
        private final TextView tvTime, tvService, tvUser, tvOffice;
        Button cancel_button;
        public ViewHolder(View view) {
            super(view);
            tvTime = view.findViewById(R.id.text_view1);
            tvService = view.findViewById(R.id.text_view2);
            tvUser = view.findViewById(R.id.text_view3);
            tvOffice = view.findViewById(R.id.text_view4);
            cancel_button = view.findViewById(R.id.cancel_button);
        }
        public TextView getTvTime() { return tvTime; }
        public TextView getTvService() { return tvService; }
        public TextView getTvUser() { return tvUser; }
        public TextView getTvOffice() { return tvOffice; }
    }
    public recycler_adapter(List<String> serviceList, List<String> timeList,
                            List<String> userList, List<String> officeList,
                            boolean admin, String currentUser, String auth) { 
        this.ServiceList = serviceList;
        this.TimeList = timeList;
        this.UserList = userList;
        this.OfficeList = officeList;
        this.isAdmin = admin;
        this.currentUserLogin = currentUser;
        this.authData = auth; 
    }
    @Override
    public ViewHolder onCreateViewHolder(ViewGroup viewGroup, int viewType) {
        View view = LayoutInflater.from(viewGroup.getContext())
                .inflate(R.layout.recycler_item, viewGroup, false);
        return new ViewHolder(view);
    }
    @Override
    public void onBindViewHolder(ViewHolder viewHolder, final int position) {
        viewHolder.getTvTime().setText(TimeList.get(position));
        viewHolder.getTvService().setText(ServiceList.get(position));
        if (OfficeList != null && OfficeList.size() > position && OfficeList.get(position) != null && !OfficeList.get(position).isEmpty()) {
            viewHolder.getTvOffice().setVisibility(View.VISIBLE);
            viewHolder.getTvOffice().setText("Отделение: " + OfficeList.get(position));
        } else {
            viewHolder.getTvOffice().setVisibility(View.GONE);
        }
        if (isAdmin && UserList != null && UserList.size() > position) {
            viewHolder.getTvUser().setVisibility(View.VISIBLE);
            viewHolder.getTvUser().setText("Пользователь: " + UserList.get(position));
        } else {
            viewHolder.getTvUser().setVisibility(View.GONE);
        }
        viewHolder.cancel_button.setOnClickListener(new View.OnClickListener() {
            @Override
            public void onClick(View v) {
                String[] dateTimeParts = TimeList.get(position).split(" / ");
                String timeStr = dateTimeParts[0];
                String dateStr = dateTimeParts[1];
                String hour = timeStr.split(":")[0];
                String minute = timeStr.split(":")[1];
                String day = dateStr.split("\\.")[0];
                String month = dateStr.split("\\.")[1];
                String year = dateStr.split("\\.")[2];
                String userToDelete;
                if (isAdmin && UserList != null && UserList.size() > position) {
                    userToDelete = UserList.get(position);
                } else {
                    userToDelete = currentUserLogin;
                }
                String req = year + "|" + month + "|" + day + "|" + hour + "|" + minute + "|" + userToDelete;
                System.out.println("Удаление: " + req);
                System.out.println("AuthData: " + authData);
                if (server_connection.remove_ticket(authData, req).equals("true")) {
                    ServiceList.remove(position);
                    TimeList.remove(position);
                    if (UserList != null && !UserList.isEmpty()) {
                        UserList.remove(position);
                    }
                    if (OfficeList != null && !OfficeList.isEmpty()) {
                        OfficeList.remove(position);
                    }
                    notifyItemRemoved(position);
                    notifyItemRangeChanged(position, ServiceList.size());
                    Toast.makeText(v.getContext(), "Талон удалён", Toast.LENGTH_SHORT).show();
                } else {
                    Toast.makeText(v.getContext(), "Ошибка удаления", Toast.LENGTH_SHORT).show();
                }
            }
        });
    }
    @Override
    public int getItemCount() {
        return ServiceList.size();
    }
}
